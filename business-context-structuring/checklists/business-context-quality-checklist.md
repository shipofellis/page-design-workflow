# Business Context Quality Checklist

Use this checklist before returning output.

## Scope control

- [ ] The output only structures business context.
- [ ] The output does not generate page strategy.
- [ ] The output does not generate page sections.
- [ ] The output does not generate page copy.
- [ ] The output does not generate wireframes.
- [ ] The output does not generate UI directions.

## Product clarity

- [ ] The product can be understood by a person outside the team.
- [ ] The product category is identified or marked unknown.
- [ ] The core job-to-be-done is identified or marked unknown.
- [ ] Product maturity is captured if provided.

## User clarity

- [ ] Primary target user is identified or marked unknown.
- [ ] Secondary users are separated if present.
- [ ] User, buyer, evaluator, and influencer are not incorrectly collapsed.
- [ ] Awareness level is captured if known.

## Pain and use case quality

- [ ] Pain points are specific, not generic.
- [ ] Use cases include trigger, action, and desired outcome when available.
- [ ] Unsupported pain points are marked as assumptions.

## Value proposition

- [ ] Core value proposition connects user pain, product capability, and user outcome.
- [ ] Supporting values are separated from the primary value.
- [ ] Vague claims are not treated as confirmed differentiators.

## Competition and differentiation

- [ ] Direct competitors are captured if provided.
- [ ] Indirect alternatives or current workarounds are captured if relevant.
- [ ] Differentiators include credibility level.

## Trust and objections

- [ ] User objections are included or marked missing.
- [ ] Trust assets are included or marked missing.
- [ ] The implications of objections for page design are captured.

## Business and measurement

- [ ] Primary business objective is clear.
- [ ] Secondary business objective is captured if provided.
- [ ] Success metrics are included or marked missing.
- [ ] Metrics connect to the page goal.

## Constraints

- [ ] Brand constraints are captured if provided.
- [ ] Technical constraints are captured if provided.
- [ ] Legal/compliance constraints are captured if provided.
- [ ] Timeline, content asset, accessibility, or localization constraints are captured if provided.

## Uncertainty handling

- [ ] Missing information is explicit.
- [ ] Assumptions are labeled as assumptions.
- [ ] Risks of assumptions are stated.
- [ ] Validation needs are stated where relevant.

## Next step readiness

- [ ] Readiness status is one of: ready, partially_ready, blocked.
- [ ] The reason is specific.
- [ ] Required action is practical.
