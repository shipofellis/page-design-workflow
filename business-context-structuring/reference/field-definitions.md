# Business Context Document Field Definitions

## product_overview

A concise description of what the product is, what category it belongs to, and what problem it addresses.

Include:

- Product category
- Core job-to-be-done
- Current product maturity if known
- One-sentence product explanation

Do not include a full marketing page narrative.

## target_users

The primary and secondary user groups relevant to the page.

Capture:

- Segment name
- Role or persona
- Context of use
- Awareness level
- Decision role if known

If unknown, mark explicitly.

## user_pain_points

Specific problems, frictions, costs, anxieties, inefficiencies, or unmet needs that make the product relevant.

Good pain points are specific and user-centered.

Weak: “Users need productivity.”
Better: “Operations managers spend hours manually reconciling repeated workflow exceptions across disconnected tools.”

## user_use_cases

Concrete situations where the product is used or evaluated.

Use cases should include:

- Trigger situation
- User action
- Desired outcome

## core_value_proposition

The central promise of value the page must communicate.

It should connect:

- User pain
- Product capability
- Desired outcome

## key_features

Product capabilities that may need to appear on the page.

Do not list every feature. Focus on features that support the value proposition or reduce user objections.

## competitive_context

Known alternatives the user might compare against.

Include:

- Direct competitors
- Indirect alternatives
- Existing user workaround
- “Do nothing” option if relevant

## differentiation

How the product is meaningfully different from alternatives.

Differentation should be specific, credible, and relevant to the target user.

## user_objections

Reasons the user may hesitate to act.

Common categories:

- Trust
- Cost
- Complexity
- Switching cost
- Security
- Time to value
- Lack of proof
- Fit for use case

## trust_assets

Evidence available to support credibility.

Examples:

- Customer logos
- Testimonials
- Case studies
- Usage numbers
- Security certifications
- Demo video
- Product screenshots
- Founder/team credibility
- Press or analyst coverage

## business_objectives

The business outcomes the page should support.

Examples:

- Waitlist signup
- Trial signup
- Demo request
- Paid conversion
- Activation
- Feature adoption
- Lead qualification

## success_metrics

How page performance will be evaluated.

Examples:

- CTA click-through rate
- Signup conversion rate
- Demo request rate
- Scroll depth
- Form completion rate
- Bounce rate
- Qualified lead rate

## constraints

Confirmed limitations or requirements affecting the page.

Examples:

- Brand constraints
- Legal or compliance requirements
- Technical implementation limits
- Launch deadline
- Design-system requirements
- Accessibility requirements
- Localization requirements

## assumptions

Working assumptions made because the user did not provide enough information.

Each assumption should be phrased as a hypothesis, not fact.

## missing_context

Information not provided that may be needed by downstream steps.

Each missing context item should specify why it matters.
